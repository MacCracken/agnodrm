## Findings

### F-7 — `fuse_extract_field` doesn't unescape `\040` whitespace in /proc/mounts

**Severity:** Informational (correctness, not security).
**Location:** `src/fuse.cyr:46-90` (`fuse_extract_field`).
**Class:** Parser confusion / mount-source escaping.

`/proc/mounts` encodes embedded whitespace in field values as `\040` (octal 040 = 0x20 = space). `fuse_extract_field` splits fields on raw space (32) / tab (9) without first unescaping `\040`, so a mount whose source path contains a space (e.g. `mount.nfs //host/share with space /mnt`) parses as two fields and shifts subsequent fields.

**Mitigation already in place:** agnosys's `fuse_validate_mountpoint` rejects relative paths and `..`, so a malicious mount source can't escape the validation by exploiting field-shift. The bug surfaces only as wrong mount metadata, not as a security boundary failure.

**Recommendation:** Add `\040` → space unescape pass. Track as a V1.1.x correctness patch (pair with H-2 + D-3 in V1.1.14 if scope permits, else its own slot).

### F-8 — Bootloader cmdline danger-flag list is conservative

**Severity:** LOW.
**Location:** `src/bootloader.cyr:362-374` (`bootloader_danger_init`).
**Class:** Allowlist freshness.

Current dangerous-flag set (case-insensitive exact match, not substring):

```
init=/bin/sh, init=/bin/bash, init=/bin/dash, single,
emergency, rd.break, ...
```

The set targets classic single-user-shell escape patterns. **Not in the set:**

- `lockdown=none` (or `lockdown=integrity`) — disables / weakens kernel lockdown.
- `module.sig_enforce=0` — disables module signature checking.
- `efi=disable_early_pci_dma` — DMA-protection escape on EFI systems.
- `noapparmor` / `enforcing=0` — disables AppArmor / SELinux.
- `apparmor=0`, `selinux=0`, `audit=0` — same family.
- `slab_nomerge=0`, `init_on_alloc=0`, `init_on_free=0` — heap-hardening downgrades.

**Recommendation:** Extend the danger-flag set. Several of these are exact-match (`audit=0`); a few would need prefix-match (`init=` covers any custom init path). Worth a follow-up PR with the full audit-canon list. Track as a V1.1.x slot or fold into V1.1.14 if the scope is small.

### F-9 — Subprocess outbuf static-data hazard relies on `dmverity_run_capture` honoring `buf_len - 1`

**Severity:** LOW.
**Location:** `src/dmverity.cyr:293, 397` (two `var outbuf[4096]` sites).
**Class:** Buffer-byte vs buffer-elements confusion / null-terminator assumption.

Both sites call `dmverity_run_capture(args, &outbuf, 4095)` — passing `4095` not `4096`, leaving 1 byte for the trailing null. The subsequent `strlen(&outbuf)` is safe iff `dmverity_run_capture` writes at most 4095 bytes and either explicitly null-terminates or `memset(&outbuf, 0, 4096)` (which both call sites do prior to the run) leaves the final byte as 0.

This is correct under current `dmverity_run_capture` behavior, but the contract is implicit. If anyone refactors `dmverity_run_capture` to consume the full `buf_len` argument without reserving the trailing byte, `strlen` reads off the end of the buffer.

**Recommendation:** Add an explicit `store8(&outbuf + 4095, 0)` after the `dmverity_run_capture` call (or make `dmverity_run_capture` document & enforce its own null-termination contract). Track as a V1.1.x patch — XS effort.

### F-10 — `secureboot_byte_at` correctly accounts for the 4-byte EFI attribute header

**Severity:** LOW (no defect — verification noted for the audit record).
**Location:** `src/secureboot.cyr:75` (`secureboot_byte_at`).
**Class:** EFI attribute offset handling.

The fn reads `load8(buf + 4 + index)`. The +4 is the EFI runtime-attribute prefix (UEFI Spec §8.2). Verified consistent across all uses of `secureboot_byte_at` in `src/secureboot.cyr` (no callers bypass via raw `load8`).

**No action required.** Recorded for audit traceability.

## Cross-cutting verification

### C-1 — Argv-only exec (no `sys_system` / shell)

**Status:** ✓ verified (0 sites in src/).
**Method:** Internal review's `grep -rn "sys_system" src/*.cyr` returned 0.
**Coverage:** All 7 subprocess-using modules (audit, journald, luks, dmverity, tpm, certpin, udev, netns, bootloader, fuse) use `exec_vec` / `exec_capture` / `run_capture` exclusively. Per ADR-001.

### C-2 — Path validation rejects `../` and relative paths

**Status:** ✓ verified for primary attack surface.
**Tests covering:**
- audit: `audit_rule_validate` rejects `../` and relative — `fw_trav` and `fw_rel` integration assertions.
- fuse: `fuse_validate_mountpoint` enforces absolute paths.
- mac: `mac_default_profile` validates agent_type charset (alpha + `-_`).
- update: `update_validate_version` (CalVer) rejects malformed.

### C-3 — Numeric-input parser bounds

**Status:** ✓ verified.
- `update_parse_version` rejects non-digit, range-checks year (2024-2100).
- `audit_rule_syscall` is just an integer accessor — no parser surface.
- `tpm_pcr_read` validates index 0-23 explicitly.
- No integer overflow paths surfaced in the review (Cyrius signed-i64 arithmetic; agnosys field semantics fit comfortably in i64).

### C-4 — Cstring null-check coverage in setters

**Status:** ✓ verified at use-site level.
The `#derive(accessors)`-generated `*_set_*` setters store the pointer raw without null checks (correct — null is a valid sentinel). Use-sites that read these pointers always handle null:
- The 5 hand-rolled `_<mod>_emit_cstr_or_null` helpers explicitly handle `c == 0`.
- `mac_profile_new` initializes `selinux_ctx` and `apparmor_name` to 0 (null) and the corresponding read sites check before `strlen`.
- `certpin_info_new` zero-initializes all fields via `memset`; readers handle null.

No defect found.

### C-5 — Read patterns relying on `strlen` of partial reads

**Status:** ✓ verified safe.
Reviewed sites:
- `dmverity.cyr`: covered under F-9 (buf_len-1 pattern is correct, contract is implicit).
- `ima.cyr`: uses `slice<u8>` indexing (V1.1.11 canonical) — bounds explicit, no `strlen` of partial read.
- `audit.cyr`: parses via length-prefixed `nlmsghdr`, not `strlen`.
- `pam.cyr`: line parser uses explicit length cursors, not `strlen`.

### C-6 — Buffer safety / `var buf[N]` static-data hazard

**Status:** ✓ verified safe (3 sites; all 4 KB).
From the internal review's H-1: `dmverity_format`, `dmverity_status`, and `ima_get_status` are the 3 fn-scope `var buf[4096]` sites. All three were verified to copy extracted bytes to a heap allocation before returning (`memcpy` to `alloc(N)`-allocated cstring or struct field). No static-buf refs escape to the caller.

The CI `Security Scan` job (`.github/workflows/ci.yml:172-218`) auto-flags any new `var buf[≥4 KB]` sites as warnings and any `var buf[≥64 KB]` as fail.

### C-7 — Pointer validation

**Status:** ✓ verified at module-API level.
All public fns taking pointer arguments validate before deref:
- Atomic-write/copy in `update.cyr` rejects null `path` / `src` / `dst` arguments.
- `audit_handle_fd` returns negative for invalid handles, callers check.
- The Cyrius `Result` discipline (Ok / Err) is the structural type-system answer to "did this work?" — there's no untyped int-as-error pattern in agnosys src/.

### C-8 — Known CVE class review

Per the [CVE landscape doc](2026-05-09-cve-landscape.md) §A–N, all 17 module-bound interfaces have known historical CVE classes documented and agnosys's mitigations cited. Open hardening surface (this audit's findings F-7 through F-10) covers the residue.
